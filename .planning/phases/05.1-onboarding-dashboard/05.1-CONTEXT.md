# Phase 5.1: Onboarding Dashboard — Context

**Gathered:** 2026-03-23
**Status:** Ready for planning
**Source:** In-conversation design session (Vanguard ADR/IA HTML reference)

<domain>
## Phase Boundary

Replace the existing Overview tab (`/customer/:id`) with a fully dynamic React onboarding status dashboard. The dashboard displays per-customer onboarding data drawn entirely from PostgreSQL — no hardcoded content. Data is entered fresh in-app (not imported from existing context docs by default). New YAML sections (`onboarding_phases`, `integrations`) are added to the context doc by a migration/import script when available; in-app edits write back to YAML. The old "Overview" React component is removed entirely.

</domain>

<decisions>
## Implementation Decisions

### Technology & Rendering
- Dynamic React component (not a skill-generated iframe or static HTML)
- Live PostgreSQL data via existing REST API patterns (no GraphQL)
- New DB tables: `onboarding_phases`, `onboarding_steps`, `integrations`
- Reuse existing `risks` and `milestones` tables for those sections
- TanStack Query v5 for data fetching (matches project-wide pattern)

### Dashboard Sections (matching Vanguard ADR/IA reference design)
1. **Sticky header** — customer name, animated progress ring (% complete based on steps), executive summary text, desired outcomes, target go-live
2. **Onboarding Phases** — collapsible cards per phase; each card lists steps with status badges (not-started / in-progress / complete / blocked), owner, dependencies, append-only update notes; filter by status and search by name
3. **Integration Tracker** — card grid of tools; each card shows: tool name, category, color badge, 4-stage pipeline bar (Not Connected → Configured → Validated → Production) + Blocked state; editable notes inline
4. **Risks & Blockers** — read from existing `risks` table (project_id scoped); cards showing severity, description, mitigation
5. **Milestone Timeline** — read from existing `milestones` table; vertical timeline with due-date and status

### Data Entry & Editing
- Onboarding phases are **static labels** (user does not add/remove phases); phases come from the template data
- Steps are editable: status badge click cycles not-started → in-progress → complete → blocked; owner is editable inline; update notes are append-only (new text appended with timestamp, not overwritten)
- Integration tracker fields editable: status (cycles through 4 stages + blocked), notes (inline textarea)
- Risks and milestones are read-only in this phase (edited via existing views)

### Source of Truth & YAML Round-Trip
- Data entered in-app is the source of truth in PostgreSQL
- `onboarding_phases` YAML section in context doc → import script seeds `onboarding_phases` + `onboarding_steps` tables when run
- `integrations` YAML section in context doc → import script seeds `integrations` table
- In-app edits write back to the YAML file (same pattern as existing YAML editor)
- Migration script is a one-time seed; running again is idempotent (ON CONFLICT DO NOTHING)

### API Endpoints
- `GET /api/projects/:id/onboarding` — phases + steps
- `PATCH /api/projects/:id/onboarding/steps/:stepId` — update step status/owner/notes
- `GET /api/projects/:id/integrations` — integration list
- `PATCH /api/projects/:id/integrations/:integId` — update status/notes
- Reuse existing `/api/projects/:id/risks` and `/api/projects/:id/milestones`

### Database Schema
```sql
onboarding_phases (id, project_id, name, display_order, created_at)
onboarding_steps (id, phase_id, project_id, name, description, status, owner, dependencies text[], updates jsonb[], display_order, created_at, updated_at)
integrations (id, project_id, tool, category, status, color, notes, display_order, created_at, updated_at)
```
`status` for steps: 'not-started' | 'in-progress' | 'complete' | 'blocked'
`status` for integrations: 'not-connected' | 'configured' | 'validated' | 'production' | 'blocked'

### Reference Design
HTML file: `/Users/jmiloslavsky/Downloads/Vanguard_ADR_IA_Onboarding_Dashboard (1).html`
Visual elements to replicate: sticky header + progress ring, collapsible phase cards, color-coded status badges, integration grid cards with pipeline bar, risk/milestone cards.

### What is NOT in scope for 5.1
- Adding/removing onboarding phases (static set per project)
- Editing risks or milestones (existing views handle this)
- Automatic sync on YAML file change (manual import script only)
- Phase 5.2 time tracking (separate phase)

</decisions>

<specifics>
## Specific Ideas

- Progress ring: calculated as `completed_steps / total_steps * 100`, animated CSS ring
- Step status cycling: single click on badge → next status in cycle; no modal needed
- Update notes: append `{ timestamp, text }` to `updates` JSONB array; render as a simple log under the step
- Integration pipeline bar: 5 segments (4 stages + blocked), highlight active stage
- Collapsible phases: default open for in-progress, closed for complete
- Filter bar: "All / Not Started / In Progress / Complete / Blocked" pills + search input
- Import script location: `scripts/import-onboarding.ts` — reads YAML, upserts to DB
- YAML write-back: same `yaml.dump()` approach as existing YAML editor, `sortKeys: false, lineWidth: -1, schema: yaml.JSON_SCHEMA`

</specifics>

<deferred>
## Deferred Ideas

- Adding/removing integrations in-app (currently seeded via import script or manual DB insert)
- Gantt chart / date-based timeline for phases
- Email/Slack notifications on step completion
- Phase 5.2: Time Tracking (separate inserted phase)

</deferred>
