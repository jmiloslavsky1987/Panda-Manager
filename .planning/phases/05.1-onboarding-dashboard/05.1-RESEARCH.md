## Summary

Phase 5.1 replaces the existing read-only `overview/page.tsx` (an RSC rendering workstreams + milestones) with a fully interactive onboarding status dashboard backed by three new PostgreSQL tables. The implementation follows established patterns from prior phases: Drizzle ORM schema + manual SQL migration, REST API routes under `app/api/`, a Client Component using `useState`/`fetch` (NOT TanStack Query — see below), and YAML write-back via the project's `serializeProjectToYaml()` utility.

The most complex new elements are: (1) the JSONB `updates` array for append-only step notes — a new pattern in this codebase; (2) the `dependencies text[]` column on `onboarding_steps` — the only array column type in the schema so far; (3) YAML write-back from an API route (not a migration script), which must integrate `lib/yaml-export.ts` and the project's workspace path from settings. The import script at `scripts/import-onboarding.ts` follows the existing `migrate-local.ts` pattern with `ON CONFLICT DO NOTHING` idempotency.

The reference HTML (`Vanguard_ADR_IA_Onboarding_Dashboard (1).html`) contains a complete working implementation of the visual design including status CSS variables, progress ring SVG, integration pipeline bar, and collapsible phase cards. All visual patterns can be translated directly into Tailwind classes within the Next.js component.

**Primary recommendation:** Build the new page as a single `'use client'` component (`OnboardingDashboard`) that fetches all data on mount via `fetch()` + `useState`, mirrors the DraftsInbox component pattern (not the modal pattern), and uses `router.refresh()` for post-PATCH UI refresh on non-optimistic updates.

## Common Pitfalls

### Pitfall 1: JSONB Array Handling in Drizzle

**What goes wrong:** Pushing to a JSONB array in a PATCH handler requires reading the existing value first (same as the `mitigation_append` pattern in risks), then appending, then setting the whole array. You cannot do JSONB array push atomically in Drizzle's `update().set()` without a raw SQL expression.

**Why it happens:** Drizzle's `.set()` replaces the column value; it does not have a built-in `jsonb_array_append` helper.

**How to avoid:** Use Drizzle's `sql` template tag for the JSONB update:
```typescript
import { sql } from 'drizzle-orm'
// Append a new update entry to the updates JSONB array:
await db.update(onboardingSteps)
  .set({
    updates: sql`${onboardingSteps.updates} || ${JSON.stringify([newEntry])}::jsonb`,
    updated_at: new Date(),
  })
  .where(eq(onboardingSteps.id, stepId))
```

**Warning signs:** If PATCH tests show only one update ever persisting (overwriting prior entries), this pitfall was hit.

### Pitfall 2: text[] Array Column in Drizzle

**What goes wrong:** Drizzle's `.array()` column requires `drizzle-orm/pg-core`'s `text` with `.array()` chained. When inserting rows, you must pass a JavaScript array (not a string). The `defaultValue` in Drizzle for `.array()` is `[]` (empty array literal), not `'{}'`.

**Why it happens:** PostgreSQL stores `text[]` as `{}` format but Drizzle's ORM layer expects JS arrays in input/output.

**How to avoid:** Pass `dependencies: []` for new rows without dependencies. When reading back, Drizzle returns a JS string array.

**Warning signs:** TypeScript compile errors on `dependencies` field typing, or "invalid input syntax for type array" from PostgreSQL.

### Pitfall 3: Enum Values with Hyphens

**What goes wrong:** PostgreSQL enum values like `'not-started'` and `'in-progress'` (with hyphens) are valid in PostgreSQL but uncommon. Drizzle's `pgEnum` handles them, but the TypeScript type inferred by Drizzle will be a string union with hyphens — which is valid TypeScript. No special handling needed.

**Why it happens:** Most existing enums in this project use underscores (`in_progress`, `not_started`). The CONTEXT.md decision is to use hyphens to match the reference HTML `data-status` attributes. This is intentional.

**How to avoid:** Be consistent — use hyphens in both the Drizzle schema declaration AND in any string comparisons in components.

### Pitfall 4: Import Path Depth in Nested API Routes

**What goes wrong:** The new routes live at `app/api/projects/[projectId]/onboarding/route.ts` — which is 6 directories deep from `bigpanda-app/`. The relative import to `db/index.ts` needs to be `../../../../../db` (5 levels up to `bigpanda-app/`, then down into `db/`). This is exactly the issue noted in STATE.md for Phase 5-03: "Stream route at runs/[runId]/stream/ is one extra directory level deep — import paths to db/ require 6 ../ not 5 (corrected during Task 1 verification)."

**How to avoid:** Count levels carefully when writing new route files. Or use the `@/` alias (if configured in tsconfig) — check if `@/` is configured in the project's tsconfig.

**Warning signs:** "Cannot find module" TypeScript errors at compile time.

### Pitfall 5: YAML Write-Back Overwriting Non-Onboarding Sections

**What goes wrong:** When writing back to the project YAML, a naive approach reads the file, rebuilds a fresh document using `buildYamlDocument()` (which uses `REQUIRED_TOP_LEVEL_KEYS`), and overwrites. This would lose any custom sections not in the required keys list (e.g., `onboarding_phases`, `integrations`).

**How to avoid:** Instead of rebuilding the entire document from DB, parse the existing YAML frontmatter, mutate only the `onboarding_phases` and `integrations` keys, then re-serialize. Use `parseYaml()` to read, update the object in-place, use `serializeProjectToYaml()` to write.

### Pitfall 6: Status Cycle Order Must Be Consistent

**What goes wrong:** If the frontend cycles `not-started → in-progress → complete → blocked → not-started` but the backend validation rejects sequences, or the cycle order differs between the badge click handler and the import script default, UX breaks.

**How to avoid:** Define the cycle array once as a constant shared by both the component and any status-display logic:
```typescript
const STEP_STATUS_CYCLE = ['not-started', 'in-progress', 'complete', 'blocked'] as const
```

### Pitfall 7: Sticky Header Conflict with WorkspaceTabs

**What goes wrong:** The reference HTML's `.header` is `position: sticky; top: 0` on the outermost container. In this app, the `WorkspaceTabs` nav is already `sticky top-0 z-10`. Adding another sticky element inside the `p-6 overflow-y-auto` scroll container will stack them — likely the desired UX (the dashboard header sticks within the content area as the user scrolls). But `z-index` conflicts are likely.

**How to avoid:** The onboarding header sticks within the scrollable `flex-1 overflow-y-auto` div (child of WorkspaceLayout). Assign it `z-index: 5` (below WorkspaceTabs' `z-10`) so it does not overlap the tab bar. Use Tailwind `sticky top-0 z-[5]`.
