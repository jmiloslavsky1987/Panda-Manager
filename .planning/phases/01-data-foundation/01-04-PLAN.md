    2. `parseYaml(content: string): unknown` — calls `yaml.load(content, { schema: yaml.JSON_SCHEMA })`. The JSON_SCHEMA prevents boolean coercion (yes/no → true/false). Use this for ALL incoming YAML parsing in the migration script.

    4. `buildYamlDocument(project: { customer: string }, sections: Record<string, unknown>): string`:
       - Build an object with keys in REQUIRED_TOP_LEVEL_KEYS order
       - For each key: `doc[key] = sections[key] ?? []` (empty array for missing keys — never omit)
       - Key mapping: YAML 'status' → `sections.overall_status ?? sections.status ?? ''`
       - Key mapping: YAML 'history' → `sections.history ?? []` (maps from engagement_history records)
       - Serialize with `serializeProjectToYaml(doc)`
       - Return: `\`---\n${yamlContent}---\n\n# ${project.customer} — BigPanda PA 3.0 Project Context\n\``

    Also export a `ProjectSections` type interface with optional fields for all 9 keys plus `overall_status`.

    Note on Cowork compatibility: the existing `server/services/yamlService.js` uses `schema: yaml.JSON_SCHEMA` in the DUMP call too, but RESEARCH.md confirms this is only needed at parse time. Use the settings above (no schema at dump time) — this matches the byte-for-byte output that Cowork expects.
