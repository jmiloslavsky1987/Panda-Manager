  <action>
The `searchAllRecords()` function in `lib/queries.ts` has a `dateBounds()` helper (or inline SQL) that appends date range conditions to each UNION arm. Three arms use `::text` cast on `timestamptz` columns, which produces `'2026-03-25 10:00:00+00'` format — not comparable to the `'YYYY-MM-DD'` strings sent from the UI.

**Read the file first** to locate the exact lines. Then make the following targeted changes:

1. Find the `outputs` UNION arm. Change the date column reference from `op.created_at::text` to `to_char(op.created_at, 'YYYY-MM-DD')` in the dateBounds WHERE clause for this arm.

2. Find the `onboarding_steps` UNION arm. Change `os.updated_at::text` to `to_char(os.updated_at, 'YYYY-MM-DD')`.

3. Find the `integrations` UNION arm. Change `i.updated_at::text` to `to_char(i.updated_at, 'YYYY-MM-DD')`.

Other UNION arms that use native TEXT date fields (`a.due`, `r.last_updated`, `kd.date`, `eh.date`, `t.due`, `te.date`) are already in ISO `YYYY-MM-DD` format — leave those unchanged.

If `dateBounds()` is a helper function that generates a SQL fragment, the fix is in how the column expression is passed — not in the helper itself. Apply `to_char()` at the call site for these 3 timestamp columns, or update the helper to accept a pre-formatted expression.

Do NOT change the `replace(/'/g, "''")` sanitization that already protects `from`/`to` values — that is correct as-is.

After the fix: setting `to=2020-01-01` in the search UI should return zero modern records (they are all from 2026). Setting `from=2020-01-01` should not exclude any records.
