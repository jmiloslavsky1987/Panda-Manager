## Summary

Phase 5.2 adds a 12th "Time" workspace tab to every customer workspace. The tab shows a filterable table of time entries (date, hours, description) with a header total, plus add/edit/delete via a modal and CSV export. All patterns are identical to those already established in Phase 5.1 (Onboarding Dashboard) and Phase 3 (Stakeholders, Actions editing). No new packages are needed; the only new infrastructure is a single migration file and four API routes.

The scope is narrow and the codebase patterns are mature. The highest-complexity decision is already resolved: modal-based add/edit (matching the Actions and Stakeholders pattern) is the correct approach given the existing shadcn Dialog infrastructure, and browser-side CSV export via `Blob` + anchor click is the simplest and most consistent approach with this app's architecture (no file-system interaction needed).

**Primary recommendation:** Follow the Phase 5.1 pattern exactly — thin RSC page shell delegates to a `'use client'` component that owns all fetch/state; four API routes at `/api/projects/[projectId]/time-entries/`; one migration file `0006_time_tracking.sql`.

### Pattern 6: Browser-Side CSV Export
**What:** Client-side function builds CSV string from entries state, creates a Blob, triggers download via temporary anchor click
**When to use:** TIME-03 — simplest approach for single-user local app; no new route needed

```typescript
// Standard browser pattern — no library needed
function exportCSV(entries: TimeEntry[], projectName: string) {
  const header = 'date,hours,description,project name'
  const rows = entries.map(e =>
    [e.date, e.hours, `"${e.description.replace(/"/g, '""')}"`, `"${projectName}"`].join(',')
  )
  const csv = [header, ...rows].join('\n')
  const blob = new Blob([csv], { type: 'text/csv' })
  const url = URL.createObjectURL(blob)
  const a = document.createElement('a')
  a.href = url
  a.download = `time-entries-${projectName}-${new Date().toISOString().slice(0,10)}.csv`
  a.click()
  URL.revokeObjectURL(url)
}
```

### Pattern 7: Drizzle Schema + Migration
**What:** `timeEntries` table in `schema.ts`, manual SQL in `0006_time_tracking.sql`
**When to use:** Established pattern — drizzle-kit is not available (see STATE.md 01-02 decision); all migrations written manually

```typescript
// Source: bigpanda-app/db/schema.ts pattern (live codebase)
export const timeEntries = pgTable('time_entries', {
  id: serial('id').primaryKey(),
  project_id: integer('project_id').notNull().references(() => projects.id),
  date: text('date').notNull(),               // ISO date 'YYYY-MM-DD' — TEXT consistent with rest of schema
  hours: text('hours').notNull(),             // stored as TEXT decimal like '1.5' — consistent with schema convention
  description: text('description').notNull(),
  created_at: timestamp('created_at').defaultNow().notNull(),
  updated_at: timestamp('updated_at').defaultNow().notNull(),
})
```

**Note on hours data type:** The schema uses TEXT for date-like fields throughout. However, hours is a true numeric value needed for summation. Two valid approaches:
- `text` — consistent with schema convention; parse with `parseFloat()` in JS; avoids Postgres numeric type complexity
- `numeric(5,2)` in SQL (mapped to `text` in Drizzle or cast at query level) — accurate decimal storage

**Recommendation:** Use `text` in Drizzle (matching schema convention) but store as ISO numeric string (e.g. `"1.5"`). Parse with `parseFloat()` for totals. This avoids introducing a new type pattern.

### Anti-Patterns to Avoid
- **Inline row editing in a `<table>`:** The Actions tab originally tried this and hit DOM nesting issues with `<span>` inside `<tr>`. Use modal (shadcn Dialog) like ActionEditModal does.
- **Using TanStack Query:** Project decision (MEMORY.md) says plain `fetch() + useState + useEffect`. Do not introduce TanStack Query even though it's listed in project memory — it was apparently superseded.
- **Using `drizzle-kit generate` for migration:** Not available in this environment (STATE.md 01-02). Write SQL manually.

## Common Pitfalls

### Pitfall 1: Hours Summation with Floating Point
**What goes wrong:** `0.1 + 0.2 = 0.30000000000000004` — displayed total is ugly
**Why it happens:** JavaScript IEEE 754 float arithmetic
**How to avoid:** Use `toFixed(2)` on the displayed total: `totalHours.toFixed(2)` — not on individual values, only on the sum for display
**Warning signs:** Total shows unexpected decimal places in UI

### Pitfall 2: Date Filter Off-by-One
**What goes wrong:** Date range filter using `>=` and `<=` on TEXT fields works correctly only if dates are stored as ISO `YYYY-MM-DD` strings (lexicographic sort matches date sort)
**Why it happens:** TEXT comparison is lexicographic; ISO dates happen to be lexicographically sortable
**How to avoid:** Store dates as `YYYY-MM-DD` ISO strings (validated server-side). Never store as `MM/DD/YYYY`.
**Warning signs:** Filter includes or excludes boundary dates unexpectedly

### Pitfall 3: Missing `project_id` in RLS context
**What goes wrong:** API route queries `time_entries` without setting `app.current_project_id` session variable; RLS policy blocks the query returning empty results
**Why it happens:** All project-scoped tables require `SET LOCAL app.current_project_id = N` inside a transaction (see STATE.md Phase 03-09)
**How to avoid:** Wrap every DB query in `db.transaction()` with `tx.execute(sql.raw(...))` preamble, exactly as in the onboarding route
**Warning signs:** GET returns empty array even when entries exist in DB

### Pitfall 4: Tab URL Collision with Existing Routes
**What goes wrong:** Segment `time` conflicts with existing route or Next.js reserved segment
**Why it happens:** Not likely but worth verifying — no existing `/time` route in this codebase
**How to avoid:** Confirm no existing `app/customer/[id]/time/` directory before creating. (Confirmed: not present in current file listing.)
**Warning signs:** 404 or wrong page renders when navigating to Time tab

### Pitfall 5: `useEffect` stale closure on filter change
**What goes wrong:** Date range filter state changes don't trigger re-fetch
**Why it happens:** `useEffect` dependency array missing `fromDate` and `toDate`
**How to avoid:** Include all filter state variables in the `useEffect` dependency array
**Warning signs:** Changing date filter doesn't update the table

### Pitfall 6: CSV with commas/newlines in description
**What goes wrong:** Unescaped commas or newlines in description field break CSV parsing
**Why it happens:** CSV format requires quoting fields containing the delimiter
**How to avoid:** Always wrap `description` in double-quotes and escape internal double-quotes: `"${str.replace(/"/g, '""')}"`
**Warning signs:** Imported CSV has wrong column count in Excel

```typescript
// Source: pattern from bigpanda-app/app/api/stakeholders/[id]/route.ts
export async function DELETE(
  _req: NextRequest,
  { params }: { params: Promise<{ projectId: string; entryId: string }> }
) {
  const { projectId, entryId } = await params
  const numericProjectId = parseInt(projectId, 10)
  const numericEntryId = parseInt(entryId, 10)

  await db.transaction(async (tx) => {
    await tx.execute(sql.raw(`SET LOCAL app.current_project_id = ${numericProjectId}`))
    await tx.delete(timeEntries).where(
      and(eq(timeEntries.id, numericEntryId), eq(timeEntries.project_id, numericProjectId))
    )
  })
  return NextResponse.json({ ok: true })
}
```

## Open Questions

1. **Should hours be stored as TEXT or NUMERIC(5,2) in Postgres?**
   - What we know: All date-adjacent fields in schema.ts use TEXT. But hours is a true numeric value.
   - What's unclear: Whether strict decimal precision matters (TIME-03 CSV export shows raw value anyway)
   - Recommendation: Use TEXT in Drizzle (consistent with schema convention); validate as parseable decimal in Zod; compute sum with parseFloat + toFixed(2) in JS. This avoids a new column type pattern and keeps migration simple.

2. **Where does project name come from for CSV export (TIME-03)?**
   - What we know: `projects.name` and `projects.customer` are available in the DB. The RSC page can pass `project.name` as a prop to the client component.
   - What's unclear: Which field — `name` or `customer`? The requirement says "project name".
   - Recommendation: Pass `project.customer` (the human-readable company name) from the RSC page shell to the client component. It is already available from `getProjectWithHealth()` in the workspace layout.

3. **Should date range filter be server-side (API query param) or client-side (filter on loaded data)?**
   - What we know: Actions tab uses URL searchParams for server-side filtering; OnboardingDashboard fetches all data client-side. Time entries are expected to be low volume (hundreds, not thousands).
   - Recommendation: Server-side via query params (`?from=&to=`) in the GET API route. This keeps the pattern consistent with Actions and makes the filter work correctly even for large date ranges without loading all data.
