## Summary

Phase 19.1 replaces the discovery scanner's MCP-only approach with a dual-path adapter system. For each source (Slack, Gong, Glean, Gmail), if org-level REST credentials exist in `settings.json` (or per-user Gmail token in DB), the scanner calls the REST API directly. The MCP path remains intact as a fallback. This makes the scanner usable by any team member, not just Claude Code sessions.

The two most technically involved parts are: (1) the Gmail OAuth 2.0 flow — a 3-step roundtrip (redirect → consent → callback → token storage) using the Google `googleapis` Node.js client, and (2) the Slack search limitation — `search.messages` requires a **user token** (`xoxp-`), not a bot token. The CONTEXT.md says "Slack bot token" but this only works with `conversations.history` (channel-specific, requires knowing channel IDs). The planner must decide: either use `conversations.history` + bot token (scoped to specific channels), or use `search.messages` + user OAuth token (global search). This is the primary open question.

All four REST API call patterns are verified from official sources. The settings extension (add `source_credentials` key to `AppSettings`) follows the exact pattern used for `mcp_servers`. The DB migration adds one table (`user_source_tokens`) using the project's established raw SQL + Drizzle schema pattern.

**Primary recommendation:** Use `conversations.history` with a bot token for Slack (simpler, no additional OAuth) and allow the user to configure a set of monitored channel names/IDs. Glean global token requires `X-Glean-ActAs: user@company.com` header — store admin email alongside token.

## Critical Finding: Slack Token Type

**The CONTEXT.md refers to a "Slack bot token" for `search.messages`.** This is incorrect per Slack's official API docs:

> `search.messages` requires a **user token** (`xoxp-`), not a bot token (`xoxb-`). Using a bot token returns `token_type_not_allowed`.

**Impact on planning:**
- Option A: Use `conversations.history` + bot token — requires knowing channel IDs; fetch from configured channel list in settings; bot must be in each channel
- Option B: Use `search.messages` + user OAuth token — more flexible but requires another OAuth flow (Slack OAuth, not just an API key)

**Recommendation for planner:** Default to Option A (bot token + `conversations.history`). The CONTEXT.md "bot token" intent aligns with Option A. Store monitored channel IDs alongside the token in `source_credentials`. If the user only provides a channel name (e.g., `#project-updates`), the adapter calls `conversations.list` to resolve the ID first.

```typescript
const oauth2Client = new google.auth.OAuth2(

  process.env.GOOGLE_CLIENT_SECRET,

return Response.redirect(authUrl);

// app/api/oauth/gmail/callback/route.ts — GET: exchange code for tokens
const { tokens } = await oauth2Client.getToken(code);
// tokens.refresh_token, tokens.access_token, tokens.expiry_date, tokens.id_token
// Decode id_token for user email, then upsert into user_source_tokens
```

### Pattern 7: Gmail API with googleapis Client (Token Refresh)
**What:** Use `googleapis` OAuth2 client with stored refresh_token; library auto-refreshes access token.
**When to use:** Gmail scanning.
```typescript
// Source: https://googleapis.dev/nodejs/googleapis/latest/
import { google } from 'googleapis';
const auth = new google.auth.OAuth2(clientId, clientSecret, redirectUri);
auth.setCredentials({ refresh_token: storedRefreshToken });
// googleapis auto-refreshes access token when expired
const gmail = google.gmail({ version: 'v1', auth });
const listResp = await gmail.users.messages.list({
  userId: 'me',
  q: `"${projectName}" after:${sinceDate}`,
  maxResults: 20,
});
```

### Anti-Patterns to Avoid
- **Calling `search.messages` with a bot token:** Returns `token_type_not_allowed`. Use `conversations.history` with a bot token instead.
- **Not storing `prompt: 'consent'` in Gmail OAuth:** Without this, Google only issues a refresh_token on the first authorization; subsequent auth flows won't return a refresh_token.
- **Hardcoding Glean `instance` name in the adapter:** The instance URL (e.g., `bigpanda-be.glean.com`) must come from stored settings; it varies per org.
- **Storing Gong `baseUrl` as hardcoded default:** Gong base URL is region-specific (`https://us-6852.api.gong.io`); must be admin-configured or discovered from the API response.
- **Masking org credentials the same way as MCP keys:** The existing `/api/settings` GET masks `apiKey` in `mcp_servers`. The new `/api/settings/source-credentials` GET should also mask stored secrets — show only last 4 chars or a boolean `connected: true`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Gmail token refresh | Custom timer / polling to refresh access tokens | `googleapis` OAuth2Client with `setCredentials({ refresh_token })` | Library handles expiry detection and silent refresh automatically; handles edge cases like network failure during refresh |
| OAuth state forgery prevention | Custom state string stored in session | Standard `state` parameter via `crypto.randomUUID()` stored in httpOnly cookie for callback validation | CSRF attack vector if state is not verified |
| Gong pagination | Custom loop tracking offsets | Use Gong `cursor` field in response — include in next request's `filter.cursor` | Gong uses cursor-based pagination; offset pagination doesn't exist in their API |
| Glean pagination | Custom offset tracking | Use Glean `cursor` in response body for next page | Same pattern |

**Key insight:** The only library addition needed is `googleapis` for Gmail. All other REST APIs use native `fetch` with straightforward auth headers.

## Common Pitfalls

### Pitfall 1: Slack `search.messages` vs `conversations.history` Token Mismatch
**What goes wrong:** The CONTEXT.md mentions "Slack bot token" alongside `search.messages`. These are incompatible. Using a bot token with `search.messages` returns HTTP 200 with `ok: false` and `error: "token_type_not_allowed"`.
**Why it happens:** Slack's `search.messages` is a user-context endpoint (searches the installing user's accessible history). Bot tokens have no user-context.
**How to avoid:** Implement `SlackAdapter` using `conversations.history` (not `search.messages`). Store optional channel IDs alongside the bot token in `source_credentials.slack.channels`. If channels array is empty, the adapter returns empty results with a warning prompting the user to configure channels.
**Warning signs:** Response body `ok: false, error: "token_type_not_allowed"` or `missing_scope`.

### Pitfall 2: Gmail Refresh Token Not Returned on Repeat Auth
**What goes wrong:** Google only sends `refresh_token` on the first consent. If a user disconnects and reconnects, the re-auth flow doesn't return a refresh_token, leaving the DB row stale.
**Why it happens:** Google's default behavior when `access_type: 'offline'` without `prompt: 'consent'` — Google assumes the refresh token already exists.
**How to avoid:** Always include `prompt: 'consent'` in the Gmail OAuth authorization URL. This forces re-consent and guarantees a fresh refresh_token.
**Warning signs:** `tokens.refresh_token` is `undefined` in the callback handler.

### Pitfall 3: Glean Global Token Missing `X-Glean-ActAs`
**What goes wrong:** Requests with a global token but no `X-Glean-ActAs` header fail with 403 Forbidden.
**Why it happens:** Global tokens are multi-user; Glean requires knowing which user's permissions to apply.
**How to avoid:** Store an `actAsEmail` field alongside the Glean token in settings. When the header is missing, the adapter should warn and skip rather than returning misleading search results.
**Warning signs:** HTTP 403 from Glean search endpoint.

### Pitfall 4: Gong Base URL is Region-Specific
**What goes wrong:** Hardcoding `https://api.gong.io` fails because Gong uses regional subdomains (e.g., `https://us-6852.api.gong.io`).
**Why it happens:** Gong's infrastructure is regionalized; the base URL is shown in the user's Gong Admin settings page.
**How to avoid:** The `source_credentials.gong` schema must include a `baseUrl` field. The Settings UI should prompt for it with instructions to find it in Gong Admin > Settings > Ecosystem > API.
**Warning signs:** HTTP 302 redirect or connection timeout.

### Pitfall 5: Missing `since` Conversion for Slack
**What goes wrong:** `conversations.history` accepts `oldest` as a Unix timestamp (float string), not an ISO date string. Passing ISO 8601 silently returns all messages.
**Why it happens:** Slack's API docs specify Slack "epoch time" (seconds since epoch, not ms).
**How to avoid:** `SlackAdapter` must convert `since` (ISO string) to Unix epoch seconds: `String(new Date(since).getTime() / 1000)`.
**Warning signs:** Scanner returns far more messages than expected (no date filter applied).

### Pitfall 6: Discovery Scanner `DiscoveryScanParams` Shape Change
**What goes wrong:** `runDiscoveryScan` currently accepts `mcpServers: MCPServerConfig[]`. Adding adapters changes the required inputs. The call site in `app/api/discovery/scan/route.ts` must be updated too.
**Why it happens:** The scan route passes `mcpServers` directly from `MCPClientPool`. After refactor, it must also read `source_credentials` from settings and `user_source_tokens` from DB to pass to the scanner.
**How to avoid:** Plan the scan route update explicitly. The scan route (step 2 in route.ts) becomes: read settings → read user tokens from DB → pass both to `runDiscoveryScan`.
**Warning signs:** TypeScript compile error in route.ts after scanner signature change.

### OAuth State Cookie Pattern (CSRF protection)
```typescript
// app/api/oauth/gmail/route.ts
const state = crypto.randomUUID();
const authUrl = oauth2Client.generateAuthUrl({
  access_type: 'offline',
  scope: ['https://www.googleapis.com/auth/gmail.readonly'],
  prompt: 'consent',
  state,

response.headers.set('Set-Cookie',

return response;

const cookieState = request.cookies.get('oauth_state')?.value;

if (!cookieState || cookieState !== queryState) {
  return Response.json({ error: 'Invalid OAuth state' }, { status: 400 });
}
```

## Open Questions

1. **Slack: Bot token + channel list vs. user token + `search.messages`**
   - What we know: Bot tokens cannot use `search.messages`. `conversations.history` requires channel IDs.
   - What's unclear: Does the team want keyword-search across all Slack history, or is scanning specific project channels sufficient?
   - Recommendation: Build `SlackAdapter` with `conversations.history` + configurable channel list. Add a note in the Settings UI: "Add channel IDs to scan (e.g. C01234567). Bot must be invited to each channel."

2. **`user_id` for `user_source_tokens` table**
   - What we know: This is currently a single-user app (no auth system).
   - What's unclear: CONTEXT.md mentions `user_id` in the table — but there is no user identity system in the app.
   - Recommendation: Use `'default'` as a hardcoded `user_id` sentinel. The Gmail callback stores tokens as `user_id='default'`. This keeps the schema forward-compatible if multi-user auth is added later.

3. **Required environment variables**
   - What we know: Gmail OAuth needs `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REDIRECT_URI`.
   - What's unclear: Are these already in `.env.local` or does the admin need to provision a Google Cloud project?
   - Recommendation: Wave 0 task should include a README-level note listing required env vars and how to create Google Cloud credentials. The Settings page should show a clear "not configured" state if env vars are missing.
