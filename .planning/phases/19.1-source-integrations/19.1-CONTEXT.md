# Phase 19.1: Source Integrations — Context

**Gathered:** 2026-03-26
**Status:** Ready for planning
**Source:** Conversation context (architectural gap discovered during Phase 19 execution)

<domain>
## Phase Boundary

The discovery scanner (Phase 19) calls external sources (Slack, Gmail, Glean, Gong) to fetch project-related communications. The current implementation tries to use MCP beta API calls, which requires each user to have MCP server URLs and API tokens configured in `~/.bigpanda-app/settings.json`. This works for Claude Code sessions but NOT for team members accessing the web app directly.

This phase builds proper source connectivity so any team member can use the discovery scanner:
- Slack, Gong, Glean: shared org-level credentials stored in the app's Settings UI (one set per org)
- Gmail: per-user OAuth (each person connects their own Google account)
- MCP remains available as an optional alternate backend (keeps existing functionality)
- Discovery scanner chooses REST API when credentials are configured, MCP as fallback

**Out of scope for this phase:** Calendar integration, other sources beyond the four, fine-grained per-user Slack/Gong/Glean ACLs (shared tokens are sufficient for now).

</domain>

<decisions>
## Implementation Decisions

### Credential Storage
- Slack bot token, Gong API key, Glean API token: stored server-side in app settings (database or settings.json)
- These are org-level shared credentials — one set for the whole team
- Gmail: per-user OAuth tokens stored in database (per user record) — each user connects their own Google account
- Credentials must NOT be stored in client-side code or exposed to browser

### Settings UI
- New "Source Connections" section in the existing Settings page
- Show connection status for each source: connected (green) / not connected (gray)
- For Slack/Gong/Glean: simple API key input fields with "Save" button
- For Gmail: "Connect Gmail" button that opens Google OAuth flow
- Setup instructions shown inline for each source when not connected
- Do NOT require all sources to be connected — each one is optional independently

### Gmail OAuth
- Use Google OAuth 2.0 with Gmail readonly scope (`https://www.googleapis.com/auth/gmail.readonly`)
- OAuth callback handler: `/api/oauth/gmail/callback`
- Store refresh token per user (user_id → refresh_token in DB)
- Show "Connected as [email]" when connected, "Disconnect" button available
- Clear setup instructions: what to do, why it's needed, privacy note
- Since this is a single-user or small-team app: can use a single Google OAuth app configured by admin

### Discovery Scanner Adapter Pattern
- Create `SourceAdapter` interface: `fetchContent(query: string, since: string): Promise<string>`
- Four adapters: `SlackAdapter`, `GongAdapter`, `GleanAdapter`, `GmailAdapter`
- Each adapter encapsulates direct REST API calls to the service
- `MCPAdapter` wraps existing MCP beta API calls (existing implementation)
- Scanner selects adapter based on: if REST credentials exist → REST adapter; else if MCP configured → MCP adapter; else → skip with warning
- Adapter selection logic lives in `discovery-scanner.ts`

### REST API Patterns
- **Slack**: Web API search endpoint `https://slack.com/api/search.messages` with bot token (`Bearer $SLACK_BOT_TOKEN`)
- **Gong**: REST API for call transcripts and emails filtered by contact/topic
- **Glean**: REST search API with bearer token
- **Gmail**: Gmail API v1 `messages.list` + `messages.get` with OAuth access token

### MCP Backward Compatibility
- MCP adapter still works exactly as before
- If a source has both REST credentials AND MCP configured, REST takes priority (REST is more reliable)
- No breaking changes to existing settings.json format
- Settings page shows MCP status separately (read from settings.json) — view only, not editable in UI

### Error Handling / UX
- Source-specific errors are non-fatal: scanner continues with other sources
- SSE `warning` events already handled in ScanForUpdatesButton — use these for partial failures
- If NO sources have credentials (REST or MCP), show actionable error in scan UI pointing to Settings
- Settings page shows which sources are ready to scan vs. need setup

### Database Schema
- New table: `user_source_tokens` — stores per-user OAuth tokens (user_id, source, access_token, refresh_token, expires_at, email)
- New table (or extend settings): `org_source_credentials` — stores org-level API keys (source, api_key, created_at, updated_at)
- Alternatively: store org credentials in settings.json under `source_credentials` key (simpler, consistent with existing MCP pattern)
- **Decision**: Use settings.json for org credentials (consistent pattern), DB for per-user Gmail tokens

### Claude's Discretion
- Token encryption at rest (nice to have, not required for v1)
- Slack workspace/channel filtering (can scope search to project-relevant channels)
- Gong API specific endpoints and rate limiting
- Glean API specific search parameters
- Gmail search query optimization
- Token refresh flow implementation details

</decisions>

<specifics>
## Specific Requirements

**SRC-01:** Settings UI shows "Source Connections" panel with status for Slack, Gong, Glean, Gmail
**SRC-02:** Slack bot token can be entered and saved in Settings; scanner uses it for searches
**SRC-03:** Gong API key can be entered and saved in Settings; scanner uses it for transcripts
**SRC-04:** Glean API token can be entered and saved in Settings; scanner uses it for document search
**SRC-05:** Gmail OAuth connect flow works end-to-end; per-user token stored and used for search
**SRC-06:** Discovery scanner uses REST adapters when credentials are configured
**SRC-07:** MCP adapters still work when MCP is configured in settings.json (no regression)
**SRC-08:** Users without credentials for a source see clear setup instructions, not silent failure
