- `runPass5ChangeDetection(items, projectId, anthropicClient)` function
- For each extracted item, calls `findSimilarEntities` to get top 3 candidates
- If candidates found, calls Claude to classify intent: update | close | remove | new
- Filters out 'new' intent results (only surfaces changes to existing entities)

  confidence: number; // 0.6-1.0 (filtered at entry)
  reasoning: string; // Claude's 1-2 sentence explanation
}

## Architecture Notes

### Why Run Pass 5 on allRawItems (Before Dedup)?

**Decision:** Pass 5 runs on allRawItems, not newItems (deduplicated subset).

**Rationale:**
- Change detection must see all extracted entities, even if they match existing DB records
- Dedup filtering removes items already in DB (isAlreadyIngested checks)
- But those items may represent UPDATES or CLOSURES, not just duplicates
- Example: Document says "Action X is now complete" → dedup would filter it as already-ingested → change detection would miss the closure

**Flow:**
```
Passes 1-4 → allRawItems
  ↓
Pass 5 (change detection) → proposedChanges
  ↓
Dedup filtering → newItems
  ↓
Store: staged_items_json = newItems, proposed_changes_json = proposedChanges
```

### Why Confidence Threshold 0.6?

**Trade-off:**
- Too low (< 0.5): False positives clutter UI with bad suggestions
- Too high (> 0.8): Misses legitimate updates with fuzzy wording

**0.6 chosen as sweet spot:**
- Filters out weak/ambiguous matches
- Preserves strong inference matches (scattered mentions, clear pattern)
- Explicit matches (>0.8) always included
- User can always reject if wrong

### Why Top 3 Candidates Only?

**Balance precision vs recall:**
- Similarity ranking is good but not perfect
- Top 3 gives Claude enough context to disambiguate
- More than 3 → longer prompts, slower responses, diminishing returns
- Claude can pick "none of these match" (intent: new) if top 3 are all wrong

### Why Non-Fatal Pass 5?

**Resilience pattern:**
- Pass 5 is an enhancement, not a requirement
- Extraction pipeline should complete even if change detection fails
- Common failure modes: pg_trgm not enabled, Claude API timeout, malformed JSON response
- Result: staged_items_json is always populated, proposed_changes_json may be null

### Entity Type Coverage

**Supported (8 types):**
- action, risk, milestone, stakeholder, workstream, focus_area, e2e_workflow, task

**Skipped (6 types):**
- history, decision — append-only (DB triggers prevent UPDATE/DELETE)
- architecture, team_engagement — no single primary text field for fuzzy matching
- before_state — point-in-time snapshot, no match makes sense
- weekly_focus — auto-generated ephemeral data, not user-authored

**Remaining entity types** (from extraction-types.ts, not in matcher):
- businessOutcome, team, note, team_pathway, onboarding_step, integration, wbs_task, arch_node
- Reason: Lower priority for document-driven updates (these are less frequently mentioned in status docs)
- Can be added in future if needed (follow existing switch-case pattern)
