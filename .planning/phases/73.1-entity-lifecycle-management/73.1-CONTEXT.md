# Phase 73.1 Context — Entity Lifecycle Management

**Inserted:** 2026-04-20 (after Phase 73 execution session)
**Requested by:** User during Phase 73 — triggered by real-world ingestion UAT observations

---

## Why This Phase Exists

During Phase 73 UAT, the user uploaded 4 documents and reviewed extracted entities. Two problems became clear:

1. **No update/remove path via documents.** If a user uploads a new document that says "this action is closed", "remove this section", or "update this outcome", the pipeline ignores the intent entirely. It only extracts entities as new inserts. Existing records are never touched.

2. **No manual edit/delete UI.** After ingestion, many entity types are effectively read-only. Users cannot correct bad extractions, remove duplicates that slipped through, or close completed items without direct DB access.

---

## User's Exact Requirements

### Pillar 1: Document-driven updates
- New document uploads should be able to **update, close, or remove** existing entities — not just insert new ones
- The most important example: **action items that are completed/closed** should be surfaced for approval so the user can close them in the system
- This applies across ALL entity types: actions, risks, decisions, milestones, e2e_workflows, architecture nodes, stakeholders, wbs_tasks, workstreams, focus_areas, etc.
- The user wants this to feel natural: upload a document, the system recognizes what changed, surfaces those changes for review in the modal (or a new tab if that makes more sense UX-wise)

### Pillar 2: Manual edit/delete for all entities
- Currently some entities have no edit/delete UI at all (read-only after ingestion)
- User needs to be able to:
  - Correct field values on any ingested entity
  - Delete incorrect or duplicate entities
  - Close/mark-complete action items and other stateful entities
- Should cover all tabs and sub-tabs where extracted entities are displayed

### User's priority statement
> "I want to emphasize the context changes as the next phase" — document-driven updates are higher priority than manual edit/delete, but both are wanted.

---

## Current Pipeline Behavior (What Needs to Change)

The current ingestion flow is **insert-only**:
1. User uploads document → BullMQ extraction job → Claude extracts entities
2. Entities staged in `extraction_jobs.staged_items_json`
3. User reviews in `IngestionModal` → approves
4. `POST /api/ingestion/approve` → `insertItem()` switch writes new rows to DB

**What happens today if you upload a document with update/remove intent:**
- If the updated entity looks like a duplicate → dedup catches it and **skips** it (old value preserved)
- If it looks different enough → writes a **second copy** alongside the old one
- Explicit "remove" or "close" instructions in document text → **completely ignored**, Claude just extracts whatever entities are described

---

## What Needs to Be Built

### A. Delta-aware extraction pass
- After the existing 4-pass extraction, add a **Pass 5: Change Detection**
- Claude compares extracted entities against what already exists in the DB for this project
- Outputs a list of proposed changes: `{ intent: 'update' | 'close' | 'remove', entityType, existingId, fields }`
- OR: a separate document upload mode explicitly for "update/correction" documents

### B. Change approval UI
- The `IngestionModal` (or a new tab) surfaces proposed changes for review
- User sees: "These items may have changed — approve to update"
- Separate section from new items: new inserts vs. proposed updates vs. proposed closures
- User approves/rejects each change individually

### C. Approve route — update/delete handlers
- `POST /api/ingestion/approve` currently only calls `insertItem()`
- Needs `updateItem()` and `closeItem()` handlers for each entity type
- Close = set status to completed/closed (varies by entity type)
- Remove = soft delete or hard delete (TBD during planning)

### D. Manual edit/delete UI per entity type
- Audit which tabs have edit/delete today vs. which are read-only
- Add edit modals and delete actions for all read-only entity types
- At minimum: actions (close/reopen), risks, decisions, e2e_workflows, architecture, wbs_tasks
- Actions specifically need a "mark as closed" UX that's fast and easy

---

## Entities Known to Lack Edit/Delete UI

From the Phase 73 UAT session, the user specifically called out:
- **Business Value & Expected Outcomes** (Teams tab sub-section) — no edit/delete
- **End-to-End Workflows** (Teams tab sub-section) — no edit/delete, no way to restructure sections
- Generally: most ingested entity tabs are read-only after ingestion

A full audit of which tabs have edit vs. not should be part of planning.

---

## Key Design Decisions (TBD During Planning)

1. **Separate upload mode vs. automatic intent detection?**
   - Option A: All uploads go through change detection automatically (complex, may have false positives)
   - Option B: User explicitly marks an upload as "update document" vs. "new content document" (simpler, clearer)

2. **How to match extracted entities to existing DB rows?**
   - Need a matching key per entity type (e.g., action matched by description similarity, stakeholder by email/name)
   - Fuzzy match risk: false positives could incorrectly update unrelated entities

3. **Close vs. Remove?**
   - Some entities should be closeable (actions, risks, milestones)
   - Some should be removable (duplicates, incorrect extractions)
   - Need to decide per entity type

4. **Modal vs. new tab for change review?**
   - Current modal is already complex (new items + existing items review)
   - A dedicated "Changes" tab within the modal might be cleaner
