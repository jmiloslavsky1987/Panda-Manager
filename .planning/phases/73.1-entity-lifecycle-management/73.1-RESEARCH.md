# Phase 73.1: Entity Lifecycle Management - Research

**Researched:** 2026-04-20
**Domain:** Entity CRUD operations, document-driven updates, change detection
**Confidence:** HIGH

## Summary

Phase 73.1 adds two critical capabilities currently missing from the extraction pipeline: (1) **document-driven updates** — allowing newly uploaded documents to update/close/remove existing entities, not just insert new ones, and (2) **manual edit/delete UI** — giving users direct control to correct, close, or remove entities after ingestion.

The current pipeline is insert-only. When a user uploads a document that says "Action A-123 is now closed", the extraction worker ignores the intent and treats it as a new entity (either skipped by dedup or inserted as a duplicate). Users discovered this during Phase 73 UAT when attempting to upload status updates and found no mechanism to reflect completed work.

**Primary recommendation:** Add a **Pass 5: Change Detection** to the extraction worker that compares extracted entities against existing DB records and outputs proposed updates/closures/removals. Surface these changes in a dedicated review UI (modal tab or separate approval flow). Implement PATCH/DELETE route handlers for all entity types and add edit/delete UI to currently read-only sections.

### Current Pipeline Behavior (What Needs to Change)

The current ingestion flow is **insert-only**:
1. User uploads document → BullMQ extraction job → Claude extracts entities
2. Entities staged in `extraction_jobs.staged_items_json`
3. User reviews in `IngestionModal` → approves
4. `POST /api/ingestion/approve` → `insertItem()` switch writes new rows to DB

**What happens today if you upload a document with update/remove intent:**
- If the updated entity looks like a duplicate → dedup catches it and **skips** it (old value preserved)
- If it looks different enough → writes a **second copy** alongside the old one
- Explicit "remove" or "close" instructions in document text → **completely ignored**, Claude just extracts whatever entities are described

### Pattern 1: CRUD Route Handler (PATCH + DELETE)

**What:** All entity types need consistent PATCH and DELETE handlers with auth, audit logging, and transaction safety.

**When to use:** For every entity type that appears in ingestion or manual creation flows.

**Example:**
```typescript
// app/api/{entity-type}/[id]/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { db } from '@/db';
import { {entityTable}, auditLog } from '@/db/schema';
import { eq, and, sql } from 'drizzle-orm';
import { requireProjectRole } from '@/lib/auth-server';

export async function PATCH(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;
  const numericId = parseInt(id, 10);
  if (isNaN(numericId)) {
    return NextResponse.json({ error: 'Invalid ID' }, { status: 400 });
  }

  // Auth: lookup entity's project_id first, then check membership
  const [entity] = await db.select({ project_id: {entityTable}.project_id })
    .from({entityTable})
    .where(eq({entityTable}.id, numericId));
  if (!entity) return NextResponse.json({ error: 'Not found' }, { status: 404 });

  const { session, redirectResponse } = await requireProjectRole(entity.project_id, 'user');
  if (redirectResponse) return redirectResponse;

  const body = await request.json();
  // Validate with Zod schema...

  // Read before-state for audit
  const [before] = await db.select().from({entityTable}).where(eq({entityTable}.id, numericId));

  await db.transaction(async (tx) => {
    await tx.execute(sql.raw(`SET LOCAL app.current_project_id = ${entity.project_id}`));
    await tx.update({entityTable})
      .set({ ...body, updated_at: new Date() })
      .where(eq({entityTable}.id, numericId));
    await tx.insert(auditLog).values({
      entity_type: '{entity_type}',
      entity_id: numericId,
      action: 'update',
      actor_id: session.user.id,
      before_json: before,
      after_json: { ...before, ...body },
    });
  });

  return NextResponse.json({ success: true });
}

export async function DELETE(
  _request: NextRequest,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;
  const numericId = parseInt(id, 10);
  if (isNaN(numericId)) {
    return NextResponse.json({ error: 'Invalid ID' }, { status: 400 });
  }

  const [entity] = await db.select({ project_id: {entityTable}.project_id })
    .from({entityTable})
    .where(eq({entityTable}.id, numericId));
  if (!entity) return NextResponse.json({ error: 'Not found' }, { status: 404 });

  const { session, redirectResponse } = await requireProjectRole(entity.project_id, 'user');
  if (redirectResponse) return redirectResponse;

  const [before] = await db.select().from({entityTable}).where(eq({entityTable}.id, numericId));

  await db.transaction(async (tx) => {
    await tx.execute(sql.raw(`SET LOCAL app.current_project_id = ${entity.project_id}`));
    await tx.insert(auditLog).values({
      entity_type: '{entity_type}',
      entity_id: numericId,
      action: 'delete',
      actor_id: session.user.id,
      before_json: before,
      after_json: null,
    });
    await tx.delete({entityTable}).where(eq({entityTable}.id, numericId));
  });

  return NextResponse.json({ success: true });
}
```

**Source:** Derived from existing pattern in `app/api/projects/[projectId]/e2e-workflows/[workflowId]/route.ts` (Phase 73 verified pattern).

### Pattern 3: Pass 5 — Change Detection in Extraction Worker

**What:** After 4-pass extraction completes, run a 5th pass that compares extracted entities to existing DB records and identifies potential updates/closures/removals.

**When to use:** When user uploads a document that may contain status updates, not just new entities.

**Example:**
```typescript
// worker/jobs/document-extraction.ts (add after Pass 4)

interface ProposedChange {
  intent: 'update' | 'close' | 'remove';
  entityType: EntityType;
  existingId: number;
  existingRecord: Record<string, unknown>;
  proposedFields?: Record<string, unknown>; // For updates
  confidence: number; // 0.0 - 1.0 match confidence
  reasoning: string; // Why Claude thinks this is a match
}

async function runPass5ChangeDetection(
  allExtractedItems: ExtractionItem[],
  documentText: string,
  projectId: number
): Promise<ProposedChange[]> {
  const changes: ProposedChange[] = [];

  // For each extracted item, check if it matches an existing entity
  for (const item of allExtractedItems) {
    const candidates = await findMatchingEntities(item, projectId);
    if (candidates.length === 0) continue;

    // Ask Claude: does this extracted item represent an update/closure of an existing entity?
    const prompt = `
You are reviewing an entity extracted from a document. Here is the extracted entity:

Entity Type: ${item.entityType}
Fields: ${JSON.stringify(item.fields)}
Source Excerpt: "${item.sourceExcerpt}"

Here are existing entities in the database that might match:
${candidates.map((c, i) => `[${i}] ID=${c.id}: ${JSON.stringify(c)}`).join('\n')}

Question: Does this extracted entity represent an UPDATE, CLOSURE, or REMOVAL of one of the existing entities? Or is it a NEW entity?

Respond with JSON:
{
  "intent": "update" | "close" | "remove" | "new",
  "matchedId": <existing entity ID if update/close/remove, else null>,
  "confidence": <0.0 - 1.0>,
  "reasoning": "<brief explanation>",
  "proposedFields": <if update, which fields changed>
}
`;

    const response = await anthropic.messages.create({
      model: 'claude-3-5-sonnet-20241022',
      max_tokens: 1000,
      messages: [{ role: 'user', content: prompt }],
    });

    const result = JSON.parse(response.content[0].text);
    if (result.intent !== 'new' && result.matchedId) {
      const existing = candidates.find((c) => c.id === result.matchedId);
      if (existing) {
        changes.push({
          intent: result.intent,
          entityType: item.entityType,
          existingId: result.matchedId,
          existingRecord: existing,
          proposedFields: result.proposedFields,
          confidence: result.confidence,
          reasoning: result.reasoning,
        });
      }
    }
  }

  return changes;
}
```

### Pattern 4: Fuzzy Entity Matching with PostgreSQL Trigram

**What:** Use PostgreSQL's `pg_trgm` extension to find similar entity matches based on primary text fields.

**When to use:** When matching extracted entities to existing DB records in Pass 5 change detection.

**Example:**
```typescript
// lib/entity-matcher.ts
import { db } from '@/db';
import { actions, risks, milestones } from '@/db/schema';
import { eq, sql } from 'drizzle-orm';

interface MatchCandidate {
  id: number;
  similarity: number;
  [key: string]: unknown;
}

export async function findSimilarActions(
  description: string,
  projectId: number,
  threshold = 0.7
): Promise<MatchCandidate[]> {
  // Requires: CREATE EXTENSION IF NOT EXISTS pg_trgm;
  const results = await db.execute(sql`
    SELECT
      id,
      description,
      owner,
      status,
      similarity(description, ${description}) AS similarity
    FROM actions
    WHERE project_id = ${projectId}
      AND similarity(description, ${description}) > ${threshold}
    ORDER BY similarity DESC
    LIMIT 3
  `);

  return results.rows as MatchCandidate[];
}

export async function findSimilarRisks(
  description: string,
  projectId: number,
  threshold = 0.7
): Promise<MatchCandidate[]> {
  const results = await db.execute(sql`
    SELECT
      id,
      description,
      severity,
      status,
      similarity(description, ${description}) AS similarity
    FROM risks
    WHERE project_id = ${projectId}
      AND similarity(description, ${description}) > ${threshold}
    ORDER BY similarity DESC
    LIMIT 3
  `);

  return results.rows as MatchCandidate[];
}

// Similar functions for milestones, stakeholders (match on name + email), etc.
```

**Source:** PostgreSQL `pg_trgm` extension documentation (https://www.postgresql.org/docs/current/pgtrgm.html). Verified available in PostgreSQL 12+.

### Anti-Patterns to Avoid

- **Automatic approval of changes without review:** Change detection must surface proposed updates for user approval, never auto-apply. False positives can overwrite critical data.
- **Client-side fuzzy matching for large datasets:** Use PostgreSQL trigram indexes for performance; JavaScript string distance is too slow for 100+ entities.
- **Hardcoded entity type lists:** Use exhaustive switch statements validated by TypeScript, not dynamic arrays. Ensures new entity types don't fall through silently.
- **DELETE without audit log:** All deletions MUST write to `audit_log` table before removing the row (established pattern in Phase 73).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Fuzzy string matching | Custom Levenshtein or Jaro-Winkler | PostgreSQL `pg_trgm` extension | Trigram already optimized with GIN indexes; custom JS implementation is 10-100x slower for large datasets |
| Change detection prompt engineering | Ad-hoc Claude prompts per entity type | Structured tool use with typed schema | Tool use enforces JSON schema, eliminates parsing errors; existing pattern in `document-extraction.ts` (EXTR-08) |
| Entity matching by external_id | Parse external IDs from document text | Match on primary text fields (description, name) | External IDs are human-assigned and may be inconsistent across documents; fuzzy text match is more reliable |
| Modal state management | useState for every field | React Hook Form or plain controlled inputs | For 3-5 fields, plain `useState` is simpler; React Hook Form adds complexity without benefit until 10+ fields |

**Key insight:** Entity lifecycle operations (CRUD) are standard database patterns with established conventions. The novel part is **change detection via AI** — this requires careful prompt design to avoid false positives, but should reuse existing extraction tool patterns.

## Common Pitfalls

### Pitfall 1: False Positive Matches in Change Detection

**What goes wrong:** Fuzzy matching incorrectly links a new entity to an existing unrelated entity, causing the system to propose an update that overwrites valid data.

**Why it happens:** Similarity thresholds too low (e.g., 0.5) or matching on generic text (e.g., "Implement authentication" matches multiple actions).

**How to avoid:**
- Set conservative similarity threshold (≥ 0.7 for trigram matching)
- Always show top 3 candidates to Claude and let it pick the best match with reasoning
- Surface ALL proposed changes to user with before/after comparison — never auto-apply
- For high-risk entities (key_decisions, engagement_history), disable change detection entirely (append-only by design)

**Warning signs:** User reports "wrong entity got updated" or "data disappeared after upload."

### Pitfall 2: Missing DELETE Route Protection

**What goes wrong:** DELETE handler doesn't check if user has permission on the entity's project, allowing cross-project deletion via URL manipulation.

**Why it happens:** Forgetting to lookup `project_id` from the entity before calling `requireProjectRole()`.

**How to avoid:**
- ALWAYS lookup entity's `project_id` in first query before auth check
- Pattern: `SELECT project_id FROM {table} WHERE id = {id}` → `requireProjectRole(project_id, 'user')`
- Never trust `projectId` from URL params alone — validate against entity's FK

**Warning signs:** API returns 200 but entity from different project gets deleted; audit logs show unexpected cross-project deletes.

**Example from existing code:**

### Pitfall 3: Edit Modal Without Optimistic Updates

**What goes wrong:** User clicks Save, modal closes, but UI doesn't update until page refresh. Feels broken.

**Why it happens:** Component doesn't optimistically update parent state before API call completes.

**How to avoid:**
- Pattern: `onUpdate(optimistic)` → `onOpenChange(false)` → `fetch()` → `onUpdate(saved)` or `onUpdate(original)` on error
- Use `router.refresh()` only as fallback; prefer controlled state updates for instant feedback
- Show spinner during save but don't block modal close

**Warning signs:** User reports "nothing happened after clicking Save" or "had to refresh to see changes."

### Pitfall 4: Append-Only Tables with Edit UI

**What goes wrong:** User tries to edit an `engagement_history` or `key_decisions` entry, but the update fails with database trigger error.

**Why it happens:** These tables have `enforce_append_only` triggers that prevent UPDATE/DELETE by design (audit trail requirement).

**How to avoid:**
- Audit which tables have append-only triggers (check schema.ts comments)
- For append-only entities: hide edit/delete buttons entirely, show "(permanent record)" label
- If correction needed: add "addendum" feature (insert new row with reference to original) instead of edit

**Warning signs:** 500 error in console with trigger violation message; user confused why edit button does nothing.

**Append-only tables (from schema.ts):**
- `engagement_history` — DB trigger prevents UPDATE/DELETE
- `key_decisions` — DB trigger prevents UPDATE/DELETE

### PATCH Handler with Auth and Audit
```typescript
// Source: app/api/projects/[projectId]/e2e-workflows/[workflowId]/route.ts (Phase 73)
export async function PATCH(
  request: NextRequest,
  { params }: { params: Promise<{ projectId: string; workflowId: string }> }
) {
  const { projectId, workflowId } = await params;
  const numericProjectId = parseInt(projectId, 10);
  const numericWorkflowId = parseInt(workflowId, 10);
  if (isNaN(numericProjectId) || isNaN(numericWorkflowId)) {
    return NextResponse.json({ error: 'Invalid projectId or workflowId' }, { status: 400 });
  }

  const { session, redirectResponse } = await requireProjectRole(numericProjectId, 'user');
  if (redirectResponse) return redirectResponse;

  const body = await request.json();
  const updates: Record<string, unknown> = {};
  if (body.team_name !== undefined) updates.team_name = body.team_name;
  if (body.workflow_name !== undefined) updates.workflow_name = body.workflow_name;

  if (Object.keys(updates).length === 0) {
    return NextResponse.json({ error: 'No fields to update' }, { status: 400 });
  }

  const [before] = await db.select().from(e2eWorkflows)
    .where(and(eq(e2eWorkflows.id, numericWorkflowId), eq(e2eWorkflows.project_id, numericProjectId)));
  if (!before) return NextResponse.json({ error: 'Not found' }, { status: 404 });

  try {
    const result = await db.transaction(async (tx) => {
      await tx.execute(sql.raw(`SET LOCAL app.current_project_id = ${numericProjectId}`));
      const updated = await tx.update(e2eWorkflows)
        .set(updates)
        .where(and(eq(e2eWorkflows.id, numericWorkflowId), eq(e2eWorkflows.project_id, numericProjectId)))
        .returning();
      await tx.insert(auditLog).values({
        entity_type: 'e2e_workflow',
        entity_id: numericWorkflowId,
        action: 'update',
        actor_id: session.user.id,
        before_json: before as Record<string, unknown>,
        after_json: { ...before, ...updates } as Record<string, unknown>,
      });
      return updated;
    });
    if (result.length === 0) return NextResponse.json({ error: 'Not found' }, { status: 404 });
    return NextResponse.json({ workflow: result[0] });
  } catch (err) {
    console.error('PATCH error:', err);
    return NextResponse.json({ error: 'Internal server error' }, { status: 500 });
  }
}
```

## Open Questions

1. **What similarity threshold for entity matching?**
   - What we know: PostgreSQL trigram `similarity()` returns 0.0-1.0; typical thresholds are 0.6-0.8
   - What's unclear: Optimal threshold per entity type (actions vs. risks vs. stakeholders)
   - Recommendation: Start with 0.7, surface top 3 candidates to Claude for final decision with reasoning

2. **Should change detection be always-on or opt-in?**
   - What we know: User wants "natural" UX (automatic), but false positives are risky
   - What's unclear: Whether users will trust automatic change detection or prefer explicit "update mode"
   - Recommendation: Start with **opt-in** (checkbox "This document contains updates to existing items") to build confidence, move to always-on in future phase if false positive rate < 5%

3. **How to handle conflicting updates from multiple documents?**
   - What we know: User could upload 2 documents in parallel that both update the same action
   - What's unclear: Last-write-wins vs. merge conflict UI
   - Recommendation: For v1, last-write-wins (simpler); add conflict detection in future phase if users report data loss

4. **Should we add soft delete or hard delete?**
   - What we know: Audit log already records before-state before hard delete; soft delete adds `deleted_at` column
   - What's unclear: Whether users need "undo delete" or if audit log recovery is sufficient
   - Recommendation: Start with **hard delete** (simpler, matches existing pattern in e2e-workflows route); add soft delete only if users request undo feature
