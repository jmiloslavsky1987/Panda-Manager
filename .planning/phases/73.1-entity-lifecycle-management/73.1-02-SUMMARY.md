```typescript
export async function DELETE(_request, { params }) {
  // 1. Parse and validate ID
  const numericId = parseInt(id, 10);

  // 2. Lookup project_id for auth
  const [entity] = await db.select({ project_id: table.project_id })
    .from(table)
    .where(eq(table.id, numericId));
  if (!entity) return 404;

  // 3. Check project role
  const { session, redirectResponse } = await requireProjectRole(entity.project_id, 'user');
  if (redirectResponse) return redirectResponse;

  // 4. Get before-state for audit
  const [before] = await db.select().from(table).where(eq(table.id, numericId));

  // 5. Transaction: SET LOCAL → audit log → delete
  await db.transaction(async (tx) => {
    await tx.execute(sql.raw(`SET LOCAL app.current_project_id = ${entity.project_id}`));
    await tx.insert(auditLog).values({
      entity_type: 'action|risk|milestone|workstream',
      entity_id: numericId,
      action: 'delete',
      actor_id: session.user.id,
      before_json: before,
      after_json: null,
    });
    await tx.delete(table).where(eq(table.id, numericId));
  });

  return { success: true };
}
```

## Architecture Notes

### Why pg_trgm Extension?

Pass 5 (change detection) requires fuzzy entity matching to detect renamed/modified entities across document updates. pg_trgm provides:
- Trigram similarity scoring (`similarity()` function)
- GIN/GIST index support for fast similarity queries
- Threshold-based matching (configurable via `pg_trgm.similarity_threshold`)

**Use case example:** Detect that "Integrate BigPanda API" → "Integrate BP REST API" is the same action (80%+ similarity).

### Why proposed_changes_json in extraction_jobs?

Alternatives considered:
1. Separate `proposed_changes` table (normalized)
2. Store in extraction_jobs (denormalized)

**Decision:** Option 2 (extraction_jobs column)

**Rationale:**
- Proposed changes are ephemeral (user approves or dismisses, then they're gone)
- 1:1 relationship with extraction job (no need for separate table)
- Simpler cleanup (delete job = delete proposed changes)
- Aligns with existing staged_items_json pattern

### Why DELETE handlers don't call updateXlsxRow()?

**Context:** actions PATCH handler calls `updateXlsxRow()` to keep xlsx in sync.

**Decision:** DELETE does NOT call updateXlsxRow().

**Rationale:**
- Hard delete from DB = correct behavior for delete operations
- xlsx file is treated as archival/export, not source of truth
- If xlsx needs updating, user can re-export or manually delete row
- Avoids file lock errors during bulk delete operations

### Auth Pattern Consistency

All entity DELETE operations now follow Phase 73 multi-tenant isolation pattern:
1. Lookup project_id from entity (not from request body — entity owns the project association)
2. Check project membership via requireProjectRole
3. Use SET LOCAL in transaction for RLS enforcement
4. Audit log with actual session.user.id (not 'default')

This ensures:
- Users can only delete entities in projects they belong to
- Audit trail captures real user identity
- RLS triggers see correct project context
- No cross-project data leakage
