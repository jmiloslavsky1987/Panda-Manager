# Phase 73.1 Verification

## Status: gaps_found

Human testing on 2026-04-20 revealed 3 functional gaps in the Pass 5 change detection system. The verifier never ran automatically because execution halted at the human verification checkpoint.

## Gaps

### Gap 1: Pass 5 silently skips update-intent documents (critical)

**Symptom:** A document containing explicit update instructions ("Merchant cutover decision has been completed", "data pipeline outage has been mitigated") caused the extraction pipeline to create NEW entities instead of proposing updates to existing ones.

**Root cause:** The extraction passes (1-4) have no concept of "update intent" — they classify instructional text as `decision`, `history`, or `note` entity types. All three are either in `SKIP_MATCH_TYPES` (decision, history) or absent from the entity-matcher switch statement (note). Pass 5 therefore produces zero proposed changes for any document where the intent is to update/close/remove existing entities but the extraction passes mis-classify the entity type.

**Evidence:** `extraction_jobs.proposed_changes_json` was NULL for job 78 (the uploaded test document). Confirmed via: `SELECT proposed_changes_json FROM extraction_jobs WHERE id = 78;`

**Required fix:** Add a dedicated direct-intent extraction pass (Pass 0 or a pre-pass) that asks Claude: "Does this document contain explicit instructions to update, close, remove, or merge existing named entities? If yes, extract the entity names, intents, and update details directly — do not rely on subsequent entity extraction passes." This should run before Passes 1-4 and store its output separately so it doesn't collide with normal entity extraction.

---

### Gap 2: Trigram similarity threshold too strict for merge/rename scenarios (medium)

**Symptom:** The Loyalty workflow merge case ("Loyalty ADR Alert-to-Incident Workflow" + "Loyalty ADR (Production) — Alert to Remediation" → combined) was not matched by Pass 5.

**Root cause:** The extracted workflow name was "Loyalty ADR Alert-to-Remediation Workflow (Combined)" — a synthesized/combined name, not a close copy of either existing workflow. Trigram similarity scores were 0.55 and 0.49 respectively, both under the 0.7 threshold.

**Required fix:** Lower the default similarity threshold from 0.7 to 0.5 for Pass 5 fuzzy matching, and compensate by requiring higher Claude confidence (≥ 0.75 instead of 0.6). This increases recall while keeping precision via the Claude classification step.

---

### Gap 3: `business_outcome` entity type missing from entity-matcher (medium)

**Symptom:** "The 'Biggy 50-user pilot' business outcome should be removed" was extracted as a `note` entity type (no match possible). Even if it had been extracted as `business_outcome`, `findSimilarEntities()` has no case for that type and returns `[]`.

**Root cause:** `entity-matcher.ts` switch statement covers 8 types (action, risk, milestone, stakeholder, workstream, focus_area, e2e_workflow, task) but not `business_outcome`. The `business_outcomes` table exists in the schema.

**Required fix:** Add a `business_outcome` case to `findSimilarEntities()` querying `business_outcomes` table by `outcome_text` or `title` field, plus add `e2e_workflow` compound-name matching improvement.
