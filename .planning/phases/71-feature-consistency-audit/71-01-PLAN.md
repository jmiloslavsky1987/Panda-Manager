Read every source file listed in the context section above. Then write FEATURE-CONSISTENCY-AUDIT.md.

**Classification framework:**

Two-tier classification for every finding:
- **BEHAVIORAL DUPLICATION** — two features do the same thing but are implemented separately (redundant code)
- **PATTERN INCONSISTENCY** — same feature category behaves differently across equivalent areas (missing parity)

Recommendation type for every finding — pick one, do not hedge:
- "Unify to A" (name the canonical component/pattern)
- "Unify to B" (name the canonical component/pattern)
- "Create new canonical" (describe what it should be)

Tone: direct and actionable. "DecisionsTableClient is missing bulk actions that Actions, Risks, and Milestones all have — add selectedIds + bulkUpdateStatus using ActionsTableClient as the canonical pattern" not "may potentially need alignment."

"Everything is consistent" is a valid finding for any group — report should confirm correctness where it exists.

If uncertain whether something is intentionally different or a gap, state that clearly and give a directional recommendation anyway.

---

**Report structure — group by feature type, not workspace area:**

### Summary Table (top of report)

A table listing each feature group, finding count, finding type (Duplication / Inconsistency / Both / None), and a one-line severity note. Makes the scope visible at a glance before reading details.

---

### 1. Search & Global Search

Audit: SearchBar.tsx vs GlobalSearchBar.tsx.

Determine: Are these intentionally distinct (different scopes, different UX contract) or is this a duplication to collapse? Document the specific differences in scope, interaction model, and output. Give a recommendation: keep both as distinct (and state why), or merge.

---

### 2. Bulk Actions

Audit: ActionsTableClient, RisksTableClient, MilestonesTableClient, DecisionsTableClient, WorkstreamTableClient.

For each table: does it have selectedIds, checkbox column, floating bulk bar, bulkUpdateStatus function?

Identify which tables have the pattern, which lack it, and whether the absence is intentional (e.g., Workstream progress slider is a different UX contract) or an inconsistency.

Recommendation per missing table: add bulk actions using ActionsTableClient pattern, or confirm intentional exclusion with rationale.

---

### 3. Filtering & Search (Table-Level)

Audit: Which filter params (q, status, owner, from, to) each table supports. Identify gaps — does Decisions lack a status filter because decisions have no status field, or because it was never added? Verify against the DB schema.

For each filter gap: is it a missing feature or an intentional absence (no field to filter by)? Give recommendation.

---

### 4. Edit Flows (Modal vs Inline)

Audit: ActionEditModal, RiskEditModal, MilestoneEditModal, StakeholderEditModal, TaskEditModal.

Also audit ActionsTableClient inline editing (OwnerCell, InlineSelectCell, DatePickerCell used inline in table rows).

Determine: For Actions — there is both a click-to-edit modal (ActionEditModal) and inline field editing in the table row. Is this intentional dual-mode or redundant?

For Decisions — is there a DecisionEditModal? If not, how do users edit decisions? Identify the gap.

For Teams section — InlineEditModal in teams/ is a separate component from the main edit modal pattern. Is this a justified domain-specific modal or a duplicate abstraction?

---

### 5. Add / Create Modals

Audit: AddActionModal, AddDecisionModal, AddMilestoneModal, AddRiskModal.

Compare structure: Do they all use the same Dialog + form pattern? Do they differ in field layout, validation approach, submit behavior (optimistic vs wait), or error handling?

Identify which patterns have diverged and which are consistent. Recommendation: standardize on the most complete pattern if diverged.

---

### 6. Empty States

Audit: EmptyState.tsx component usage vs inline empty state patterns.

From codebase: ActionsTableClient, RisksTableClient, MilestonesTableClient, DecisionsTableClient use EmptyState component. WorkstreamTableClient uses inline `<p>` text. Some tables also have inline "No X found" text for the filtered-zero-results case.

Identify:
1. Which areas use the EmptyState component vs inline text
2. Whether the "filter zero results" state and "no records exist" state use different patterns (they should, but verify consistency)

Recommendation: define canonical usage for each case.

---

### 7. Status Enum Handling

Audit: How status values are defined and enforced for Actions, Risks, Milestones, and Tasks.

From schema.ts: actionStatusEnum is a DB pgEnum. Milestone and Risk status appear to be component-level constant arrays (MILESTONE_STATUS_OPTIONS, RISK_STATUS_OPTIONS) without corresponding DB enums.

Determine:
1. Which entities have DB-level enum enforcement vs component-level string options only
2. Whether the status value sets are consistent (e.g., do milestone statuses in the DB route handler match MILESTONE_STATUS_OPTIONS in the component?)
3. Whether missing DB enums create a risk of invalid values reaching the database

Recommendation: identify where DB enums should be added, or confirm existing constraints are sufficient.

---

### 8. API Route Patterns (Infrastructure)

Audit: Compare auth and authorization patterns across the entity API routes.

From codebase: actions/route.ts, milestones/route.ts, decisions/route.ts all use requireSession(). Check if all entity routes use requireSession() consistently, and whether any also apply requireProjectRole() for project-scoped entities.

Note: This group covers infrastructure consistency, not UI. Keep brief — one finding per pattern difference.

---

After writing all sections, conclude with:

### Findings Summary

Count of findings by type (Behavioral Duplication / Pattern Inconsistency) and by severity (High = affects data integrity or user capability, Medium = UX inconsistency, Low = code cleanliness).

2-3 sentences: overall assessment of consistency health and highest-priority areas for Phase 73 to address first.
