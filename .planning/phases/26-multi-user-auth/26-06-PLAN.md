# Plan 26-06: Email-Based Invite Flow

## Objective

Replace the admin-side password creation flow with a secure, token-based invite flow. The admin enters an email and role; the invitee receives an email with a unique link, clicks it, sets their own password, and their account is activated.

## Motivation

Admin setting a password on behalf of a user is a security anti-pattern. The invite flow ensures only the invitee knows their own credentials.

## Changes

### Schema
- Add `invites` table: `id`, `email`, `role`, `token` (unique), `invited_by` (FK users), `expires_at` (48h), `created_at`
- Manual migration: `db/migrations/0021_invites.sql`

### New: `lib/email.ts`
- Nodemailer transport from `SMTP_HOST/PORT/USER/PASS`
- `sendInviteEmail(to, inviteUrl, invitedByName)` — plain text + HTML

### Modified: `POST /api/settings/users`
- Remove `auth.api.signUpEmail()` call
- Check no existing user or pending invite for email
- Insert invite row, send email, roll back on send failure

### Modified: `GET /api/settings/users`
- Return `{ users: [...], pendingInvites: [...] }`

### New: `DELETE /api/settings/invites/[id]`
- Admin-only cancel invite

### New: `GET + POST /api/auth/invite/[token]` (public)
- GET: validate token, return `{ email, role }`
- POST: accept invite — hash password (bcrypt), insert user + credential account, delete invite

### New: `app/accept-invite/page.tsx` (public)
- Reads `?token=` from URL
- Validates on mount via GET
- Form: email (readonly), name, password, confirm
- On success: redirects to `/login`

### Modified: `components/settings/UsersTab.tsx`
- Replace inline create form with Send Invite form (email + role, no password)
- Add Pending Invites section below user table with Cancel action

### Modified: `proxy.ts`
- Add `/accept-invite` to public path allowlist
